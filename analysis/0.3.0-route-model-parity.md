# Route & Model Parity: openrelay-like 0.3.0 vs romgX/openrelay

> **⚠️ Historical report (pre-0.3.3).** This document describes the 0.3.0 codebase. For the current (0.3.3) status, refer to [PARITY_OPENRELAY.md](../PARITY_OPENRELAY.md). The content below is preserved for reference and may not reflect the current implementation.

> **Purpose**: Detailed comparison of routing and model mapping capabilities between `openrelay-like` 0.3.0 and [romgX/openrelay](https://github.com/romgX/openrelay) (latest release v0.10.48, GitHub source v0.8.3). Covers model aliases, fallback routing, and upstream model mapping.
>
> **Analysis method**: Publicly available documentation only (README, FAQ, CHANGELOG, COMMERCIAL-LICENSE). No upstream binary was run. Evidence levels: [R] = README/FAQ claim, [RL] = Release artifact, [S] = Public source, [U] = Not black-box verified.

## 2. Fallback Routing

### 2.1 Supported Strategies

| Strategy | romgX/openrelay | openrelay-like | Comparison |
|----------|----------------|----------------|------------|
| Fallback (try A → B → C) | [R] + [U] "auto-failover" | ✅ Supported via `fallback` strategy | Aligned |
| Round-robin | [R] Listed as **Pro feature** (COMMERCIAL-LICENSE) | ✅ Supported via `round_robin` strategy | **Ahead** (not gated by license) |
| Weighted routing | Not documented in upstream publicly | ✅ Supported via `weighted` strategy | Open (upstream may have it in Pro) |
| Health-aware ordering | [R] "health check" | ✅ Supported via `provider-health.js` sliding window | Aligned |

### 2.2 Failover Behavior

| Aspect | romgX/openrelay | openrelay-like | Verification |
|--------|----------------|----------------|-------------|
| Cooldown on failure | [R] Yes, 60s | ✅ 60s default cooldown (configurable via `retry.cooldownMs`) | Aligned |
| Max attempts | Not documented | ✅ Configurable via `retry.maxAttempts` | Open |
| Stream failover | [R] Not documented separately | ✅ Supported and tested (`test-provider-fallback.mjs`) | Verified |
| 429 / rate-limit aware | [R] "auto-switch" | ✅ **Implemented in 0.3.1** — basic immediate 429 failover (skip to next provider candidate). Full quota-aware usage-based routing not implemented. | **Addressed** |
| All providers fail | [R] Returns error | ✅ Returns 503 with descriptive error | Aligned |

### 2.3 Gap

- **429-aware failover (0.3.1)**: Basic immediate 429 failover was added in 0.3.1 — when an upstream returns HTTP 429, the relay breaks the inner retry loop for that provider and immediately tries the next candidate. This avoids burning through all API keys on a quota-limited provider. Full quota-aware routing (usage-based estimation before selecting a candidate) is not implemented.
- **401/403 behavior** (0.3.1): 401/403 errors are treated as non-retryable — the relay does not retry the same provider with a different key, but falls back to the next candidate in the route. This allows other providers to handle the request when one provider's key is invalid.
- **Round-robin is Pro upstream**: Our implementation is MIT-licensed, which is an advantage but not a compliance gap.
- **Verification note**: All behavior confirmed via mock upstream tests (`test-provider-fallback.mjs`, 13 assertions + `test-quota-aware-fallback.mjs`, 22 assertions). No upstream binary verification was performed.

## 3. Upstream Model Mapping

### 3.1 Model Renaming

| Aspect | romgX/openrelay | openrelay-like | Comparison |
|--------|----------------|----------------|------------|
| User model ≠ upstream model | [R] Implied by route system | ✅ Route candidate specifies both provider AND model → upstream receives mapped model | Aligned |
| Transparent renaming | [R] User requests "fast-model", upstream receives "deep-model" | ✅ Verified via mock tests | Aligned |
| Direct model match (no route) | [R] User model sent as-is | ✅ Direct match sends user model as-is | Aligned |
| Stream with mapping | [R] Not documented separately | ✅ Upstream receives mapped model in stream requests | Verified |

### 3.2 Verification Results

| Test Scenario | Result | Evidence |
|---------------|--------|----------|
| Route with mapped model (user "fast-model" → upstream "deep-model") | ✅ Upstream receives "deep-model" | `test-upstream-model-map.mjs` mock body inspection |
| Route with same-name mapping (user "fast-model" → upstream "fast-model") | ✅ Upstream receives "fast-model" | Same test |
| Direct model without route | ✅ Upstream receives user model | Same test |
| Stream with model mapping | ✅ Upstream receives mapped model in streaming body | Same test, streaming mock endpoint |

### 3.3 Gap

- **Model mapping visibility**: Our admin/status endpoint exposes route candidate models as configured. Upstream may additionally expose per-request mapping in dashboard logs — cannot confirm.
- **Dynamic model mapping**: Upstream may support per-request model override (e.g., `x-model` header). Our implementation only supports `model` field in the request body.
- **Verification note**: All mapped model names are verified by inspecting the actual HTTP body received by the mock upstream server, not by checking admin/status alone.
